# Крипто-проксі (ProxyHandler)

Node.js/Express проксі-сервер для роботи з АЦСК (акредитованими центрами сертифікації ключів) — CMP, OCSP, TSP-запити та хмарний підпис (наприклад, Вчасно, Ощадбанк тощо). Використовується разом із IIT-бібліотекою електронного підпису на фронтенді, яка сама кодує тіло запиту у base64 перед відправкою на `/ProxyHandler`.

Сервер приймає запити від браузера і перенаправляє їх до хоста АЦСК, звіреного з білим списком, захищаючи від SSRF-атак (довільних адрес призначення).

## Вимоги

- Node.js 18+ (потрібен глобальний `fetch`)
- npm

## Встановлення

npm:

```bash
npm install
```

або pnpm (якщо використовуєте):

```bash
pnpm install
```

## Налаштування

1. Скопіюйте приклад файлу оточення:

   ```bash
   cp env.example .env
   ```

2. Відредагуйте `.env` під свої потреби:

   | Змінна | Опис | За замовчуванням |
   |---|---|---|
   | `PORT` | Порт, на якому піднімається сервер | `3000` |
   | `ENABLE_TEST_MODE` | `true` — довантажує тестові ЦСК з `CAs.Test.json` на додачу до `CAs.json`; `false` — тільки бойові ЦСК (production) | `true` |
   | `ALLOWED_ORIGINS` | Список origin'ів через кому, яким дозволено користуватись проксі (браузерні адреси, а не адреси АЦСК!). Порожнє значення = дозволено з будь-якого сайту (небезпечно для production) | `http://localhost:5173` (з прикладу) |

3. Переконайтесь, що в корені проєкту є файли `CAs.json` (список бойових АЦСК) і, за потреби, `CAs.Test.json` (тестові АЦСК). Сервер читає їх при старті та формує білий список дозволених хостів.

## Запуск

npm:

```bash
npm start
```

pnpm:

```bash
pnpm start
```

Або напряму:

```bash
node server.js
```

Після старту в консолі з'явиться:

```
Крипто-проксі працює на http://localhost:3000
Ендпоінт для бібліотеки: http://localhost:3000/ProxyHandler
```

## Використання

Фронтенд (бібліотека IIT-підпису) звертається на:

```
http(s)://<host>:<port>/ProxyHandler?address=<адреса АЦСК>&contentType=<опційно>
```

- `address` — обов'язковий параметр, повна адреса кінцевого сервісу АЦСК (CMP/OCSP/TSP тощо).
- `contentType` — опційний, якщо потрібно явно задати `Content-Type` upstream-запиту (інакше визначається автоматично за шляхом).

Сервер:
- пропускає лише `http://` і `https://` адреси;
- перевіряє хост призначення за білим списком (`PROD_HOSTS` + домени з `CAs.json`, і, якщо увімкнено тестовий режим, `TEST_HOSTS` + домени з `CAs.Test.json`);
- ручну обробляє редиректи, дозволяючи перехід лише на інший хост із того ж білого списку;
- повертає відповідь у base64 з заголовком `Content-Type: X-user/base64-data`.

## Важливо для production

- Обов'язково задайте `ALLOWED_ORIGINS` — без нього проксі приймає запити з будь-якого сайту.
- Встановіть `ENABLE_TEST_MODE=false`, щоб вимкнути тестові ЦСК.
- Файл `.env` не комітиться в git (див. `.gitignore`) — не забудьте створити його на сервері вручну або через CI/CD secrets.
